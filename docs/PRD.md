## 1. 项目背景与目标

### 1.1 背景
项目已具备 4 个独立模块，每个模块单独可用：
- `distilled_characters` — 蒸馏人物语态库（端口 8767）
- `select_topic` — 选题抓取与匹配（端口 8766）
- `writing` — 对抗式文章写作，含多平台优化（端口 8788）
- `Autopublish` — 三平台自动发布（公众号 / 小红书 / 头条）

本次重构新增 1 个模块：
- `platform_scorer` — **平台评分 agent**（端口 8789）。独立服务，输入一份稿件，输出每个平台的评分+理由。

此前已尝试用一个编排器（orchestrator, 端口 8800）将 4 者串联，但因模块间契约未对齐、状态机不清晰、错误处理不统一，导致调试成本极高，决定**保留模块、重构编排层与契约层**。

### 1.2 目标
**一行话：** 一个本机长跑、以全自动为主、人可随时介入的多平台内容生产流水线。

**量化目标（MVP 达成即结）：**
- 端到端跑通连续 5 天，每天每个平台 ≥ 1 篇成功发布
- 全程无需重启进程
- 所有失败都有可读错误 + 一键干预入口

## 3. 核心使用场景

### S1 · 全自动跑全程（默认模式）
定时任务唤醒 → 抓选题 → 选题与人物匹配 → 写作（多平台版本）→ 平台评分 → 进入"待审"队列 → 超时未审则自动发布 → 记录结果。

### S2 · 用户提选题
UI 提交一个选题（标题 + 可选材料链接）→ 跳过抓取，进入"匹配人物"环节，后续同 S1。

### S3 · 审稿与改稿
稿件生成后默认进入"待审"。用户可：
- 直接通过 → 进入发布
- 修改标题/正文/封面 → 保存后通过（**不自动重新评分，需手动点 UI 上的"重新评分"按钮触发**）
- 驳回 → 任务终止
- 不操作 → 默认信任 AI，**生成后 2 小时**自动发布

### S4 · 出错干预
任意环节失败 → UI 标红 + 显示原因 → 用户可：
- 一键重试（同模块重跑）
- 跳过该平台（仅对发布环节）
- 终止任务
- 修改输入数据后重试

### S5 · 平台评分与调度
每篇稿件生成后，**平台评分 agent**（独立模块 `platform_scorer`）输出形如：
```json
{
  "wechat":      { "score": 88, "reason": "...深度长文契合公众号..." },
  "xiaohongshu": { "score": 62, "reason": "...图片占比不足..." },
  "toutiao":     { "score": 75, "reason": "...时效性强..." }
}
```
**调度规则：**
1. 默认按评分高低决定推送平台（阈值 ≥ 70 推送）
2. **每个平台保底每天 1 篇**：编排器每日 **23:00** 扫描，若某平台当日 0 篇 → 强制把"今日所有稿件中该平台评分最高的一篇"推送过去
3. 用户可在 UI 上手动覆盖（强发某平台 / 屏蔽某平台 / 重新评分）

### 4.1 做（in scope）
- ✅ 4 个现有模块原样复用
- ✅ 每个模块新增**统一契约接口**（一层 REST adapter）
- ✅ 新增编排器：状态机驱动、SQLite 持久化、任务调度、Web UI
- ✅ 新增 **`platform_scorer` 平台评分 agent**（独立模块，端口 8789）

- ❌ 评论 / 私信回复自动化
- ❌ 多用户 / 权限系统
- ❌ 云端部署 / k8s
- ❌ 移动端 App
- ❌ 实时推送通知（Server-Sent Events 等优化，MVP 用轮询）

## 5. 任务状态机

```
                       ┌─────────────┐
                       │  collected  │ ← 抓取/用户提交
                       └──────┬──────┘
                              ▼
                       ┌─────────────┐
                       │  matched    │ ← 匹配人物语态
                       └──────┬──────┘
                              ▼
                       ┌─────────────┐
                       │  writing    │ ← 写作中
                       └──────┬──────┘
                              ▼
                       ┌─────────────┐
                       │  drafted    │ ← 稿件已生成
                       └──────┬──────┘
                              ▼
                       ┌─────────────┐
                       │  scored     │ ← 平台评分已生成
                       └──────┬──────┘
                              ▼
                       ┌─────────────┐
                       │  reviewing  │ ← 待审（含 2h 超时自动发）
                       └──────┬──────┘
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
        ┌──────────┐    ┌──────────┐    ┌──────────┐
        │ rejected │    │publishing│    │ failed   │ ← 任一环节失败
        └──────────┘    └────┬─────┘    └──────────┘
                             ▼              （可重试 / 干预）
                       ┌──────────┐
                       │published │
                       └──────────┘
```

**关键规则：**
- 每个状态都持久化到 DB（重启不丢任务）
- 每次状态切换写一条 audit log
- `failed` 不是终态：可重试回到上个状态
- `rejected` 是终态

## 6. 模块契约（高层，详见设计文档）

每个外部模块必须暴露：

| 接口 | 方法 | 用途 |
|---|---|---|
| `/health` | GET | 健康检查（编排器轮询） |
| `/contract` | GET | 返回该模块的契约版本号 + schema |
| 业务端点 | POST | 输入/输出 JSON 严格遵循设计文档约定 schema |

**契约约束：**
- 所有时间字段统一 ISO 8601 + UTC
- 所有 ID 字段是字符串（UUID v4）
- 所有错误响应统一格式：`{ "error": { "code": "...", "message": "...", "details": {...} } }`
- 图片资产：写作环节生成的 OSS 临时 URL **必须在生成后立即下载并落地到本地路径**，发布环节只读本地文件

## 7. 非功能需求

### 7.1 可靠性
- 任一外部模块崩溃，编排器不崩、不丢任务
- 编排器自身崩溃后重启可恢复所有未完成任务
- 所有外部 HTTP 调用：超时（默认 30s）+ 重试（最多 3 次，指数退避）
- **任务级失败重试上限**：每个状态最多自动重试 3 次，间隔 30s / 2min / 10min（指数退避），总时长 ≤ 15 分钟；超过则任务转入 `failed`，等待人工干预

### 7.2 可观测性
- 全链路 trace_id：任务创建即生成，贯穿所有模块的日志
- 日志统一写到 `/orchestrator/logs/`（按天分文件）
- UI 提供"任务详情页"：可看每个状态的耗时、输入输出、错误堆栈

### 7.3 性能（MVP 不优化）
- 每天 ≤ 10 篇文章吞吐即可
- 单篇任务总耗时 ≤ 30 分钟（含 LLM 调用）

### 7.4 安全
- API key / cookies 存放在 `.env` + `.cookies/`，**禁止入库 git**
- 编排器 Web UI 仅监听 127.0.0.1（不暴露公网）

## 9. 验收标准（MVP "做完了"的定义）

**必须全部满足：**

| # | 标准 | 验证方式 |
|---|---|---|
| AC1 | 连续 5 天，每天每个平台 ≥ 1 篇成功发布 | 看 published 表 |
| AC2 | 5 天内编排器进程未崩溃过 | 看 launchd 日志 + uptime |
| AC3 | 所有 failed 任务都能在 UI 上看到明确错误信息 | 抽查 |
| AC4 | 用户可在任意环节看到任务当前在哪一步 | UI demo |
| AC5 | 用户可一键重试 / 跳过 / 终止任意失败任务 | UI demo |
| AC6 | 用户可手动提选题，10 分钟内进入写作环节 | E2E 测试 |
| AC7 | 平台评分 agent 对每篇稿给出 3 个平台的评分+理由 | DB 抽查 |
| AC8 | 评分调度规则正确触发：每平台保底 1 篇/天 | 看 5 天发布分布 |

## 10. 关键风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| Playwright Cookie 失效 | 高 | 发布全失败 | UI 提供 cookie 状态检测 + 失效提醒 |
| LLM API 限流 / 失败 | 中 | 写作环节卡住 | 重试 + 切换备用 model（DeepSeek ↔ Qwen） |
| OSS 图片 URL 24h 过期 | 高 | 发布时 403 | 写作环节立即下载落地，发布只读本地 |
| 公众号草稿接口异常 | 中 | 发布失败 | 健全 errcode 解析（已踩过坑） |
| 三平台账号风控 | 中 | 账号封禁 | 控制发布频率，UI 显示发布间隔 |
| 设计期遗漏的契约不一致 | 中 | 联调返工 | 设计阶段写完整契约文档+示例 |

## 11. 选题去重策略（已确认）

**MVP 方案：分层规则去重，时间窗 7 天，不上 embedding。**

### L1 · 标题归一化完全匹配
- 选题入库时，对标题做：去标点、转小写、移除中英文停用词、去多余空白
- 与近 7 天所有选题的归一化标题完全相同 → 标记重复
- 优点：零成本；缺点：改一字就漏

### L2 · 实体 + 主题词比对（兜底）
- 选题入库时调用 1 次 LLM 提取结构化字段：
  ```json
  {
    "entities": ["DeepSeek", "梁文锋", "500亿融资"],
    "topic_keywords": ["AI融资", "大模型", "中国创业"]
  }
  ```
- 与近 7 天选题逐条比对：
  - 实体集合 Jaccard ≥ 0.7 **且** topic_keywords 重叠 ≥ 1 → 标记重复
- 提取结果落 DB（`topics.entities`, `topics.topic_keywords` 字段）

### 行为
- 命中重复 → 状态置为 `duplicated`（终态），不进入写作流
- UI 上能看到"与某月某日的《XXX》重复（实体匹配 80%）"
- 用户可点"仍要写"覆盖去重判定

### 不上 L3（语义 embedding）的理由
- 单用户每天 ≤ 10 条选题，规则法已足够
- embedding 需向量存储 + 相似度索引，复杂度收益不匹配 MVP
- 规则法可解释、可调阈值、UI 友好
